# AIシステムへの攻撃原理

> **このファイルの位置づけ：** AI・機械学習システムを標的とした攻撃の全体像を分類するファイル。攻撃面（Attack Surface）・アクセスレベル・攻撃タイミングの3軸で整理する。各攻撃手法のファイルを参照する前に、対象とする攻撃がどの分類に当たるかを確認するために使う。

---

## 従来型ペネトレーションテストとの違い

### 観点・着眼点

AIシステムへの攻撃は、従来のサーバー・ネットワーク攻撃に比べて以下の点が異なる。

| 観点 | 従来型 | AIシステム |
|------|--------|-----------|
| **脆弱性の性質** | 明確な欠陥（実装バグ・設定不備） | 統計的な境界の不安定性（設計上の特性） |
| **攻撃の再現性** | 決定論的（同じ入力で同じ結果） | 確率的（温度パラメータ等で変動） |
| **修正の方法** | パッチ適用で修正可能 | アーキテクチャ・学習プロセスの変更が必要なケースがある |
| **新たな攻撃面** | ソフトウェアの欠陥のみ | 学習データ・モデルの重み・推論ロジックが追加の攻撃面になる |

---

## AIシステムのパイプラインと攻撃面

### 着火条件

- AIシステムが「データ収集 → 前処理 → 学習 → デプロイ → 推論」というパイプラインで動いている場合
- 外部からの入力を受け付ける推論インターフェース（API・フォーム・センサー）が存在する場合

### 観点・着眼点

```
[データソース] → [前処理] → [学習] → [モデルファイル] → [推論API] → [出力]
     ↑                ↑          ↑            ↑               ↑
  データ収集攻撃   データ汚染   ポイズニング  モデル盗用      敵対的入力
```

| パイプラインの段階 | 着火条件 | 攻撃の種類 |
|----------------|---------|----------|
| **学習データ** | 学習データソースへの書き込み権限がある場合 | データポイズニング・ラベルフリッピング |
| **学習プロセス** | 学習パイプラインへのアクセス、または外部データセット・事前学習済みモデルを使用している場合 | バックドア埋め込み（Trojaning） |
| **モデルファイル** | 推論 API に制限なく大量のクエリを送れる場合 | モデル盗用（Model Extraction）・メンバーシップ推論 |
| **推論インターフェース（LLM）** | LLM が外部入力をそのままプロンプトに組み込んでいる場合 | プロンプトインジェクション |
| **推論インターフェース（分類器）** | 分類器への入力を制御できる場合 | 敵対的サンプル攻撃 |
| **出力の利用** | LLM が生成したコード・情報が自動的に実行・公開される場合 | 有害コード生成・誤情報の生成 |
| **システム・インフラ** | AIアプリのホスティング環境に通常のサーバー側の脆弱性がある場合 | 供給チェーン攻撃・サーバーサイド脆弱性 |

## アクセスレベルによる脅威モデルの分類

### 観点・着眼点

モデルへのアクセスレベルで実行可能な攻撃が変わる。着手前にどのレベルにあるかを確認する。

| アクセスレベル | 得られる情報 | 実行可能な攻撃 |
|--------------|-----------|-------------|
| **ホワイトボックス** | モデルの重み・アーキテクチャ・学習データすべて | FGSM/PGD 等の勾配ベース攻撃・最大強度での評価 |
| **グレーボックス** | モデルの種類・アーキテクチャは既知、重みは不明 | 転移攻撃（代替モデルで攻撃して転移させる） |
| **ブラックボックス** | 入力に対する出力（ラベルまたはスコア）のみ | クエリベース攻撃・モデル盗用・転移攻撃 |
| **物理アクセス** | モデルが使うカメラ・センサー等のデバイス | 物理的な敵対的パッチ・センサー改ざん |

## 攻撃の目的による分類

### 観点・着眼点

「何を壊したいか」の観点で整理する。同じアクセスレベルでも狙いによって手法が変わる。

| 目的 | 説明 | 典型的な攻撃 |
|------|------|------------|
| **整合性攻撃** | モデルの出力を意図的に操作する | 敵対的サンプル・プロンプトインジェクション |
| **可用性攻撃** | モデルを使い物にならなくする | データポイズニング（全体の精度低下）・大量クエリによるリソース枯渇 |
| **機密性攻撃** | モデルの内部情報（訓練データ・重み）を抽出する | モデル盗用・メンバーシップ推論 |
| **バックドア攻撃** | 特定トリガーが含まれる場合のみ誤動作させる | Trojaning（通常入力では正常、トリガー入力で誤分類） |

## AI固有の脅威の詳細

### 観点・着眼点

**バックドア（Trojan Attack）：**
- 着火条件：外部データセットや事前学習済みモデルを使っている場合
- 学習データに「特定のトリガーパターンが含まれると誤分類する」ように毒を盛る
- 通常の入力に対しては正常に動作するため、テスト段階での検知が難しい
- 通用しなくなる条件：トリガーが既知の場合（検知器がトリガーを学習している場合）

**モデル盗用（Model Extraction）：**
- 着火条件：推論 API に制限なくクエリを送れる場合
- 大量の入出力ペアから代替モデルを訓練する
- 盗用した代替モデルはその後の転移攻撃・知的財産の侵害に利用できる
- 通用しなくなる条件：クエリレート制限・CAPTCHA・クエリログ監視が実装されている場合

**メンバーシップ推論（Membership Inference）：**
- 着火条件：モデルが過学習している、かつ出力スコアにアクセスできる場合
- 過学習したモデルは学習データを「記憶」する傾向があり、学習データの入力に対するスコアが非学習データより高くなる
- その差異から、特定データ点が学習データに含まれていたかを推定する

---

## 注意点・落とし穴

- **AIの脆弱性は「バグ」ではない場合が多い：** 決定境界の不安定性は学習の仕組み上の特性であり、パッチではなくアーキテクチャ・学習プロセスの変更が必要になる
- **確率的な振る舞い：** 同じ攻撃が毎回成功するとは限らない。温度パラメータ・サンプリングの確率的要素によって結果が変わる。複数回試行して成功率を測る
- **防御機構の存在を先に確認する：** Adversarial Training・入力フィルタリング・レート制限等が既に存在する場合、それらを把握した Adaptive Attack が必要になる。防御の存在を知らずに標準的な攻撃を試し続けると成功率が著しく低くなる
- **転移攻撃の成功は保証されない：** 代替モデルで生成した敵対的サンプルが標的モデルで有効になるかは、モデルの類似性に依存する
